1.2  Key Issues Identified
Several structural and data quality problems need addressing before or during migration:
Duplicate location tables: re_locations and re_project_locations both exist and overlap. Many projects have entries in both. Consolidate to re_project_locations.
re_project_advisors UNIQUE KEY is on project_id only — this limits a project to one advisor. This is likely a bug, not intentional.
price_unit inconsistency: Values like 'L', 'l', 'Lac', 'CR', 'Cr', 'cr', ' Cr' (with spaces) are all present. Needs normalization.
advisor_whatsapp_link stored as pre-generated URL in re_leads: this is fragile — if advisor phone changes, old rows are stale.
users table has orphan columns (astro, journel, shopping, profile_pic as TINYBLOB) from a prior non-RE product.
Mixed source_name/source_type NULLs in re_projects — scraper-ingested rows can be identified but not consistently queried.
No foreign key from re_properties to re_project_locations — properties lack a structured lat/lng association through the location model.

2.1  Why Keep 4 Services on the Same MySQL DB (For Now)
Since inventory, media, user, and CRM tables are already in the same MySQL instance and share implicit FKs (e.g. advisor_id in re_leads references users.id), splitting them into completely separate databases immediately would require either:
Removing cross-service foreign keys and accepting eventual consistency, or
Running a data migration that copies/denormalizes user data into the CRM schema.
The pragmatic approach is to logically isolate them as separate Spring Boot services that each access their own schema namespace or own set of tables within the same MySQL instance. You can then physically separate them when traffic justifies it.
The agreement-service must be on PostgreSQL from day one — it needs append-only event sourcing, JSONB for clause negotiation, and row-level locking for legal document state transitions.

4.2  n8n Workflow Map
Your existing n8n instance on Docker handles orchestration. These are the key workflows to build:
Lead Intake Flow: Webhook → Validate → Routing Config lookup → Assign Advisor → WhatsApp notification → Create followup_schedule row
Scraper Reconciliation: Cron trigger → Query re_projects_staging → Deduplicate against re_projects by RERA number → Insert/skip → Mark merge_status
Follow-up Scheduler: Cron every 15 mins → Query re_followup_schedule for due rows → Send WhatsApp/SMS via provider → Update status
Agreement Signed: Webhook from Leegality/eMudhra → Update esign_requests → Update agreement status → Notify advisor via CRM → Trigger stamp delivery
Marketing Campaign: Campaign created → Query matching lead profiles → Batch send via Bulk Message Shooter → Log to re_campaign_leads
